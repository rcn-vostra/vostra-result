# Design — `Vostra.Result` Core v1.1

**Date:** 2026-06-20 · **Status:** Draft for approval · **Branch:** `feat/vostra-results-core-v1.1`

A focused follow-up to Core v1 (merged) addressing gaps an independent review found
(`docs/superpowers/notes/2026-06-20-core-review-triage.md`) and adopting decision **B** for the
Created/Ok distinction. Purely additive + a rename; no behavioral regressions to existing passing tests
(50/50). Serialization (FR-12), the NFR benchmark/coverage gates, and early-exit (OD-3) are explicitly
**deferred** to later cycles.

## 1. Error type rename (whole-codebase)

- Abstract base `Error` → **`ErrorBase`**.
- Concrete `UnexpectedError` → **`Error`** (still `Type = ErrorType.Unexpected`, the 500 kind).
- Net shape: `abstract ErrorBase` ← { `Error` (general/unexpected), `ValidationError`, `NotFoundError`,
  `ConflictError`, `AlreadyExistsError`, `UnauthorizedError`, `ForbiddenError` }.
- Every reference to the base type updates to `ErrorBase`: implicit conversions, `Errors`
  (`IReadOnlyList<ErrorBase>`), `FirstError`, `Match`/`Switch` delegates, `MapError`, aggregation,
  `ResultSentinels`, and all tests.
- **`Error` carries an edge-only XML-doc note:** it represents an unexpected fault / boundary
  exception-translation (preserve the throwable via `CausedBy`), **not** a routine domain return — domain
  services return the typed kinds and let genuinely unexpected things throw (requirements Non-Goals).

## 2. Drop `ErrorType.Failure`

- Remove `Failure` from the enum. `Unexpected` is the single 500-class kind (produced by `Error`).
- Satisfies FR-2.2 ("Unexpected/Failure (500)" — one 500 bucket). Update the Core spec text.

## 3. Adopt B — `SuccessKind` + `Result.Created` (resolves FR-3.4 / OD-7)

- Add neutral `enum SuccessKind { Ok, Created }` (transport-neutral, symmetric with `ErrorType`; **no HTTP**).
- `Result<T>` and non-generic `Result` carry a `SuccessKind` on the success branch (default `Ok`).
  Implicit conversions and existing factories yield `Ok`. Success path stays zero-alloc (one extra byte).
- Factories: `Result.Created<T>(T value)` and non-generic `Result.Created()` (201-with-no-body case).
- Expose `SuccessKind Success { get; }` (or `bool IsCreated`) for the boundary to read. AspNetCore (parked)
  maps `Ok→200`, `Created→201`. Core itself stays HTTP-free.
- Equality: two successes are equal only if value-equal **and** same `SuccessKind`.

## 4. Non-generic `Result` combinator parity

The void result is currently inspect-only (`IsSuccess`/`Errors`/`FirstError`/`Combine`), which forces a
manual `if (IsError)` on `Delete`-style ops (a P1-flavored footgun). Add the missing surface, mirroring
`Result<T>` where it makes sense for a valueless result:

- **Inspect:** `Match<TOut>(Func<TOut> onOk, Func<IReadOnlyList<ErrorBase>,TOut> onErr)`, `MatchFirst`,
  `Switch(Action onOk, Action<IReadOnlyList<ErrorBase>> onErr)`, `SwitchFirst`.
- **Compose:** `Then(Func<Result>)`, `Then<T>(Func<Result<T>>)`, `Tap(Action onOk)`,
  `TapError(Action<IReadOnlyList<ErrorBase>>)`, `MapError(Func<ErrorBase,ErrorBase>)`,
  `Ensure(Func<bool>, ErrorBase)` / `Ensure(Func<bool>, string)` (→ `ValidationError`).
- **Async parity:** `Task<Result>` receiver overloads of the above (sync + async projection), uniform names,
  `ConfigureAwait(false)` throughout — matching the `Result<T>` matrix.

## 5. `ErrorBase` re-wrap helpers (FR-2.4 / P11; fixes the spec example)

The Core v1 design example used `Error.Prefix("…")`, which doesn't exist. Add type-preserving copy helpers
on `ErrorBase`:

- `WithCode(string code)`, `Prefix(string prefix)` (prefixes `Message`), `WithMetadata(...)`,
  `WithCausedBy(Exception)`.
- Each returns a new error of the **same concrete type**, preserving `Code`/`Type`/`CausedBy`/`Metadata`
  except the field being changed. Mechanism (plan decides): a protected virtual `CopyWith(...)` each subclass
  overrides, or convert the hierarchy to `record class` with `with`-expressions + retained custom equality.

## 6. `IEnumerable` collector (FR-9.1 completion)

- Add `Combine(this IEnumerable<Result<T>>)` → `Result<IReadOnlyList<T>>` and `Combine(this IEnumerable<Result>)`
  (accumulate all errors; success collects values in order). The `params` variants already exist.

## 7. Doc reconciliations (no code change)

- `ErrorBase.Equals` compares `(concrete type, Type, Code, Message)` and **ignores** `CausedBy`/`Metadata`
  (correct for FR-11.3 identity assertions). Update the Core v1 spec §5 text to match (it currently claims
  structural metadata compare).
- Add a comment on `SelectAsync` that `selector` should return errors-as-values, not throw.

## 8. Out of scope (deferred, recorded)

System.Text.Json converter (FR-12) → own cycle before Testing. BenchmarkDotNet zero-alloc gate (NFR-2/§9.7)
and coverage run (NFR-7) → before publishing. Early-exit `Scope`/`OrReturn` (OD-3) → revisit after Testing.
`ValueTask` matrix (FR-6.1) → deferred.

## 9. Acceptance

1. Rename complete: builds green on `net8.0;net9.0`, all existing 50 tests pass after mechanical updates,
   no public `Error` (base) remains — base is `ErrorBase`, concrete general error is `Error`.
2. `ErrorType` has no `Failure`; `Error` documents its edge-only intent.
3. `Result.Created(value)` / `Result.Created()` produce `SuccessKind.Created`; plain success is `Ok`;
   success path zero-alloc; equality accounts for `SuccessKind`.
4. Non-generic `Result` supports `Match`/`Switch`/`Then`/`Tap`/`TapError`/`MapError`/`Ensure` (sync + `Task`).
5. `ErrorBase.Prefix(...)`/`WithCode(...)` exist and preserve concrete type; the spec's example compiles.
6. `IEnumerable<Result<T>>.Combine()` collects values/errors.
7. Each change lands with usage-style tests (DX-first).
